# Слои и зависимости

Проект разделен на три слоя:
```text
Infrastructure
    |
    v
Application
    |
    v
Domain
```

Стрелка означает "может зависеть от", но не наоборот

## Domain

Папка:
```text
Domain/
```

Это предметная модель. Она не знает, как именно приложение получает запросы и где хранит данные

Сейчас здесь лежит модель задач:
- `Task`
- `TaskId`
- `TaskStatus`

`Task` - объект задачи

`TaskId` - value object для публичного UUID задачи. Он не дает создать id из строки не в UUID-формате

`TaskStatus` - enum допустимых статусов

Domain не должен импортировать
- `Infrastructure`
- HTTP request/response
- PDO
- SQLite
- JSON
- routes

## Application

Папка:
```text
Application/
```

Это слой сценариев. Он отвечает на вопрос: что приложение умеет делать?

Здесь лежат:
- use case'ы
- DTO входных данных
- application contracts
- idempotency-модель

Примеры use case'ов:
- `CreateTaskUseCase`
- `CreateTaskIdempotentlyUseCase`
- `ListTasksUseCase`
- `GetTaskUseCase`
- `UpdateTaskUseCase`
- `DeleteTaskUseCase`
- `RunIdempotentOperationUseCase`

Application зависит от Domain, но не зависит от Infrastructure

Например, use case не знает про `SQLiteTaskRepository`. Он зависит от:
```php
Application\Contracts\TaskRepositoryInterface
```

А какая реализация будет подставлена, решает DI container во внешнем слое

## Infrastructure

Папка:

```text
Infrastructure/
```

Это технический слой. Он соединяет приложение с внешним миром

Здесь лежат:
- config
- HTTP kernel
- router
- middleware
- request/response
- controllers
- request mappers
- presenters
- DI container
- SQLite repositories
- PDO connection
- migrations
- console command для миграций

Infrastructure может знать про Application и Domain, потому что она адаптирует внешний мир к внутренней модели

Примеры:
```text
HTTP JSON -> RequestMapper -> Application DTO
Database row -> TaskMapper -> Domain object
Domain object -> TaskPresenter -> API JSON
```

## Граница HTTP

HTTP-граница находится в:
```text
Infrastructure/Http/
```

Там находятся:
- middleware
- controllers
- request mappers
- presenters
- response classes
- routes

Controller не содержит SQL. Он координирует:
```text
RequestMapper
-> UseCase
-> Presenter
-> Response
```

Auth находится на HTTP-границе, потому что он работает с header'ами и HTTP-кодами

Пример:
```text
Authorization header -> BearerTokenMiddleware -> Controller
```

Application и Domain не знают про `Authorization: Bearer <token>`

CORS тоже находится на HTTP-границе. Он работает с браузерными HTTP-заголовками и preflight-запросами, а не с бизнес-правилами задач

Пример:
```text
OPTIONS request -> CorsMiddleware -> NoContentResponse 204
Response -> CorsMiddleware -> CORS headers
```

Application и Domain не знают про `Access-Control-Allow-Origin`, `OPTIONS` и browser preflight

## Граница хранения

Граница хранения находится в:
```text
Infrastructure/Persistence/
```

Repository implementation знает про SQLite и PDO

Application знает только про interface:
```text
TaskRepositoryInterface
```

Это позволяет держать SQL вне use case'ов и controller'ов.

## Где что менять

Если меняется бизнес-правило задачи, смотри `Domain/Task`

Если меняется сценарий приложения, смотри `Application/UseCase`

Если меняется входной HTTP-формат, смотри `Infrastructure/Http/RequestMapper`

Если меняется защита HTTP route, смотри `Infrastructure/Http/Middleware` и `Infrastructure/Http/Routes`

Если меняется JSON-ответ API, смотри `Infrastructure/Http/Presenter`

Если меняется SQL или схема хранения, смотри `Infrastructure/Persistence` и `Infrastructure/Database/migrations`

Если меняется сборка зависимостей, смотри `Infrastructure/DI/AppContainerFactory.php`
