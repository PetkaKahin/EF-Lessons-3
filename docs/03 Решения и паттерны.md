# Решения и паттерны

## Use Case

Use case - отдельный класс под сценарий приложения

Пример:
```text
CreateTaskUseCase
UpdateTaskUseCase
DeleteTaskUseCase
```

Controller вызывает use case, а не repository напрямую

Зачем:
- сценарий можно читать отдельно от HTTP
- сценарий можно тестировать без web-сервера
- controller остается адаптером, а не местом бизнес-логики

## DTO

DTO описывает входные данные use case

Примеры:
```text
CreateTaskInput
UpdateTaskInput
ListTasksInput
```

DTO уже не является HTTP-запросом. Его можно создать из JSON, CLI-аргументов или теста

## Request Mapper

Request mapper переводит HTTP request в DTO.

Примеры:
```text
CreateTaskRequestMapper
UpdateTaskRequestMapper
ListTasksRequestMapper
TaskIdPathMapper
```

Mapper работает с внешним форматом:
- JSON body
- query params
- path params
- headers, если они нужны конкретному действию

## Repository

Repository отделяет сценарий приложения от базы данных

Use case зависит от:
```php
Application\Contracts\TaskRepositoryInterface
```

SQLite-реализация:
```php
Infrastructure\Persistence\Task\SQLiteTaskRepository
```

Так SQL остается в Infrastructure, а Application работает с задачами как с доменными объектами

## Mapper для базы

`Infrastructure\Persistence\Task\TaskMapper` переводит:
```text
database row <-> Domain\Task\Task
```

Он нужен, потому что формат БД и формат доменной модели не обязаны совпадать

Например, в БД поле называется `created_at`, а в API оно отдается как `createdAt`

## Presenter

Presenter переводит доменный объект в API-ответ
```text
Task -> TaskPresenter -> array -> JsonResponse
```

Domain не должен решать, как поле называется в JSON. Это ответственность HTTP/API-слоя

## Dependency Injection

DI container находится в:
```text
Infrastructure\DI
```

`Container` хранит factories и singleton instances

`AppContainerFactory` описывает object graph приложения:
```text
TaskRepositoryInterface -> SQLiteTaskRepository
IdempotencyKeyRepositoryInterface -> SQLiteIdempotencyKeyRepository
TransactionManagerInterface -> PdoTransactionManager
```

Классы получают зависимости через constructor. Внутри use case'ов и controller'ов нет `new SQLiteTaskRepository()`

## Idempotency

`POST /tasks` поддерживает header:
```text
Idempotency-Key
```

Это нужно, чтобы повтор одного и того же POST-запроса не создавал вторую задачу

Поток такой:
```text
TaskController
-> CreateTaskRequestMapper
-> CreateTaskIdempotentlyUseCase
-> RunIdempotentOperationUseCase
-> CreateTaskUseCase
```

`RunIdempotentOperationUseCase`:
- резервирует key
- сравнивает hash запроса
- выполняет операцию в транзакции
- сохраняет результат
- при повторе возвращает сохраненный результат

Данные idempotency хранятся в таблице `idempotency_keys`

## Transactions

Транзакции спрятаны за интерфейсом:
```php
Application\Contracts\TransactionManagerInterface
```

PDO-реализация:
```php
Infrastructure\Database\PdoTransactionManager
```

Idempotency использует транзакцию, чтобы эти действия были атомарными:
```text
зарезервировать key
создать задачу
сохранить результат операции
```

## Error Handling

HTTP-ошибки маппятся централизованно:
```text
Infrastructure\Kernel\Application
-> Infrastructure\Kernel\ExceptionHandler
```

Контроллеры не оборачивают каждый action в `try/catch`

Примеры:
```text
JsonException -> 400
InvalidArgumentException -> 422
IdempotencyConflictException -> 409
IdempotencyInProgressException -> 409
unknown Throwable -> 500
```

## Router и Response

Router самописный и лежит в:
```text
Infrastructure\Kernel\Router
```

Он связывает HTTP method + path с handler'ом

Response-классы:
- `JsonResponse`
- `NoContentResponse`

Они отвечают за HTTP status, headers и отправку body
