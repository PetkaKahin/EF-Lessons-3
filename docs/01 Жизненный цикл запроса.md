# Жизненный цикл запроса

## 1. Вход в приложение

Запрос приходит в Nginx, дальше попадает в PHP-FPM и запускает:
```text
public/index.php
```

`index.php` делает три вещи:
- запоминает время старта для debug-заголовка `X-App-Time`
- подключает Composer autoload
- запускает `Infrastructure\Kernel\Application`

Бизнес-логики здесь нет.

## 2. Application как HTTP kernel

`Infrastructure\Kernel\Application` - центральная точка выполнения HTTP-запроса

Она:
- создает DI container через `Infrastructure\DI\AppContainerFactory`
- достает `Router`
- регистрирует routes и middleware
- создает `Request` из глобальных PHP-данных
- отправляет `Request` в router
- после `dispatch` прогоняет готовый `Response` через response middleware
- отправляет готовый `Response`

Здесь же находится общий `try/catch` для HTTP-ошибок. Если внутри приложения выбрасывается понятное исключение, `ExceptionHandler` превращает его в JSON-ответ с нужным HTTP-кодом

Важно, что response middleware выполняется после `try/catch`. Поэтому технические заголовки вроде CORS добавляются не только к успешным ответам, но и к ошибкам `400`, `401`, `403`, `422`, `500`, если middleware подходит под текущий path

Если в config включен `DEBUG`, response middleware добавляет заголовок:
```http
X-App-Time: 12.3ms
```

Он лежит в headers, а не в JSON body. Так тело API-ответа остается стабильным и не меняется из-за debug-режима

## 3. Request

`Infrastructure\Kernel\Request` - объект HTTP-запроса

Он хранит:
- HTTP method
- path
- body
- headers
- query params
- path params

Контроллеры работают с этим объектом, а use case'ы уже нет. Use case'ы получают DTO и доменные объекты

## 4. Router

`Infrastructure\Kernel\Router` выбирает handler по method и path

Примеры:

```text
GET /health       -> HealthController
POST /tasks       -> BearerTokenMiddleware -> TaskController::create
GET /tasks/{id}   -> TaskController::get
PATCH /tasks/{id} -> BearerTokenMiddleware -> TaskController::patch
DELETE /tasks/{id}-> BearerTokenMiddleware -> TaskController::delete
```

Для `/tasks/15` router добавляет path param:
```php
['id' => '15']
```

Router также умеет применять middleware к группе путей. Например CORS подключается не внутри `CorsMiddleware`, а в routes:
```php
$router->middleware($container->get(CorsMiddleware::class), '/tasks');
```

Это значит:
```text
применять CorsMiddleware к /tasks и /tasks/...
```

Сам `CorsMiddleware` не знает про `/tasks`. Он знает только, как ответить на `OPTIONS` и какие CORS headers добавить к ответу

## 5. Middleware

Middleware - это обертка вокруг handler'а route

В проекте есть два вида middleware:
- request middleware - выполняется до controller и может не пустить запрос дальше
- response middleware - выполняется после controller или после `ExceptionHandler` и может изменить готовый response

Глобально подключен `DebugHeadersMiddleware`. Он пропускает request без изменений, а на response добавляет `X-App-Time`, если в config стоит:
```php
'DEBUG' => true
```

Для изменения задач защищены:
```text
POST /tasks
PATCH /tasks/{id}
DELETE /tasks/{id}
```

Поток такой:
```text
Router
-> BearerTokenMiddleware
-> TaskController
```

`BearerTokenMiddleware` проверяет header:
```text
Authorization: Bearer <token>
```

Токен берется из `API_TOKEN` в `config.php`

Если header не передан, возвращается:
```text
JsonResponse 401
WWW-Authenticate: Bearer
```

Если token неверный, возвращается:
```text
JsonResponse 403
```

Если token верный, запрос идет дальше в controller

### CORS middleware

CORS включен для всех путей:
```text
/tasks
/tasks/...
```

Подключение находится в `Infrastructure/Http/Routes/routes.php`:
```php
$router->middleware($container->get(CorsMiddleware::class), '/tasks');
```

`CorsMiddleware` берет разрешенный frontend origin из config:
```text
APP_URL
```

И добавляет к ответам:
```http
Access-Control-Allow-Origin: http://localhost:5173
Access-Control-Allow-Methods: GET, POST, PATCH, DELETE, OPTIONS
Access-Control-Allow-Headers: Content-Type, Authorization, Idempotency-Key
```

### Preflight

Preflight - это предварительный `OPTIONS`-запрос от браузера.

Он появляется перед некоторыми cross-origin запросами. Например перед запросом с:
```text
PATCH
Authorization
Content-Type: application/json
Idempotency-Key
```

Поток для preflight такой:
```text
OPTIONS /tasks/15
  |
  v
Application::run()
  |
  v
Router::dispatch()
  |
  v
CorsMiddleware::handle()
  |
  v
NoContentResponse 204
  |
  v
Application::run()
  |
  v
Router::processResponse()
  |
  v
CorsMiddleware::processResponse()
  |
  v
ResponseWithHeaders
```

На preflight бизнес-логика задач не вызывается. `TaskController` и `BearerTokenMiddleware` тоже не вызываются, потому что `CorsMiddleware` возвращает `204` раньше

После успешного preflight браузер уже отправляет настоящий запрос:
```text
PATCH /tasks/15
```

Для обычного `/tasks`-запроса поток такой:
```text
GET /tasks
  |
  v
CorsMiddleware::handle()
  |
  v
TaskController
  |
  v
JsonResponse
  |
  v
CorsMiddleware::processResponse()
  |
  v
ResponseWithHeaders
```

Если внутри запроса случилась ошибка, поток такой:
```text
PATCH /tasks/15
  |
  v
JsonException / InvalidArgumentException / другое исключение
  |
  v
Application::run()
  |
  v
ExceptionHandler
  |
  v
JsonResponse 400/422/500
  |
  v
Router::processResponse()
  |
  v
CorsMiddleware::processResponse()
```

Поэтому frontend видит нормальную ошибку API, а не только CORS-ошибку браузера

## 6. Controller

Его задача:
```text
HTTP request -> Application input
Application result -> HTTP response
```

Например, `TaskController::create()`:
1. Читает body через `CreateTaskRequestMapper`
2. Читает header `Idempotency-Key`
3. Считает hash запроса, если idempotency включена
4. Вызывает `CreateTaskIdempotentlyUseCase`
5. Превращает `Task` в JSON через `TaskPresenter`
6. Возвращает `201 Created` и `Location`

Controller не пишет SQL и не создает `Task` вручную

## 7. RequestMapper

Request mapper переводит сырой HTTP-запрос в DTO

Пример:
```text
Request body
-> CreateTaskRequestMapper
-> CreateTaskInput
```

Mapper проверяет внешний формат: JSON object, разрешенные поля, типы полей, query params, path params

После mapper'а use case получает уже нормализованные данные

## 8. UseCase

Use case - сценарий приложения

Примеры:
- `CreateTaskUseCase`
- `CreateTaskIdempotentlyUseCase`
- `ListTasksUseCase`
- `GetTaskUseCase`
- `UpdateTaskUseCase`
- `DeleteTaskUseCase`

Use case не знает, что запрос пришел по HTTP. Его можно вызвать из controller, теста, CLI-команды или очереди.

## 9. Repository

Use case работает с контрактом:
```php
Application\Contracts\TaskRepositoryInterface
```

Контракт говорит, что приложению нужно уметь делать с задачами:
- создать
- найти по id
- получить страницу
- сохранить
- удалить

Реальная SQLite-реализация лежит во внешнем слое:
```php
Infrastructure\Persistence\Task\SQLiteTaskRepository
```

Она знает про SQL, PDO и таблицу `tasks`

## 10. Domain object

Repository возвращает доменный объект:
```php
Domain\Task\Task
```

`Task` хранит состояние задачи и правила изменения:
- `rename`
- `changeDescription`
- `changeStatus`

`TaskId` проверяет корректность id

`TaskStatus` ограничивает статус значениями `new`, `in_progress`, `done`

## 11. Presenter и Response

`TaskPresenter` переводит доменный объект в API-формат:
```text
Task -> array -> JsonResponse
```

`JsonResponse` отправляет JSON и HTTP-код

`NoContentResponse` используется для успешного удаления и возвращает `204` без body


## Пример: ошибка в запросе

```text
POST /tasks
Authorization: Bearer <token>
Body: invalid json
  |
  v
BearerTokenMiddleware
  |
  v
JsonException
  |
  v
Application::run()
  |
  v
ExceptionHandler
  |
  v
Router::processResponse()
  |
  v
JsonResponse 400
```

Контроллер не ловит эту ошибку сам. Все HTTP-маппинг ошибок собран в одном месте
